# JobMonitor

## Block 9.1 update: user-defined search tracks

The fixed `AI Automation` and `Infrastructure & Business Projects` directions are no longer product constraints. A profile now has common context plus one or more independently enabled user-defined search tracks.

- The user writes the name, desired work, roles/signals, experience, skills, tasks, locations, countries, formats, and optional growth opportunities for every track.
- Locations and work formats belong to each track, not to a single global list.
- The questionnaire has three common questions and seven questions per track; optional questions can be skipped, but the experience summary and core track description cannot.
- `profile_search_tracks` stores generic track data. Existing `profile_directions` records are migrated once to generic tracks without deleting the prior table or user profiles.
- AI receives all enabled tracks for semantic evaluation. Per-track scoring and notification categories remain Block 9.3 work.

Telegram profile commands: `/start`, `/profile`, `/directions`, `/edit_profile`. The persistent menu exposes Profile, Directions, and Edit profile.

## Block 9.3 update: track-aware AI decision and Telegram card

Every opportunity is evaluated semantically against every enabled user-created search track. The AI selects the strongest track, returns an overall percentage for that track, and assigns one delivery category:

- `priority` (`Сильное совпадение`) — a direct match supported by concrete evidence;
- `review` (`Есть смысл посмотреть`) — a plausible, adjacent, growth, or uncertain match that the user should inspect;
- `archive` (`В архив`) — no meaningful relation, an explicit hard exclusion, or an invalid/non-work listing.

The old numeric threshold is retained only for previously stored legacy analyses. New `priority` and `review` results are delivered regardless of that legacy threshold; `archive` is stored but not sent.

The Telegram card contains: category and percentage, vacancy title and company, employment type when stated, work format, location, salary only when stated, selected search track, up to three concise match reasons, up to two material risks, employer-requested application actions, a short factual application draft, source, publication date when available, and the vacancy link. The normal vacancy URL button remains. A separate copy button is not included in this iteration.

# Цель проекта

Создать production-ready Python-приложение, работающее 24/7 на VPS в Docker, которое автоматически получает новые вакансии из официальных лент Habr Career, Remote OK, We Work Remotely, Remotive, Jobicy, выбранных карьерных страниц Greenhouse и открытого API «Работа России», анализирует их с помощью AI и отправляет только наиболее подходящие варианты пользователю в Telegram.

Проект создаётся одновременно как:

- личный инструмент;
- учебный проект по Python;
- портфолио;
- основа для дальнейшего коммерческого продукта;
- универсальная платформа для подключения новых провайдеров вакансий.

# Основной сценарий работы

```text
Habr Career RSS -----+
Remote OK API -------+--> JobMonitor --> AI --> Telegram
WWR public RSS ------+         |
Remotive API --------+         |
Greenhouse boards ---+         |
Rabota Rossii API ---+         |
Jobicy API ----------+         |
                              v
                         History / Storage
```

Основной worker просыпается каждые `CHECK_INTERVAL_SECONDS`, но каждый источник
имеет собственный интервал опроса и может быть отдельно включён или отключён.
В каждом цикле приложение:

1. определяет, какие источники должны быть запущены в текущем цикле;
2. независимо обращается к разрешённым официальным лентам Habr Career, Remote OK, We Work Remotely, Remotive, Jobicy, настроенным Greenhouse-доскам и API «Работа России»;
3. получает новые или обновлённые вакансии по заданным критериям;
4. нормализует вакансии в единый внутренний формат;
5. отбрасывает уже обработанные записи;
6. анализирует каждую новую вакансию через AI;
7. получает структурированный ответ;
8. принимает решение;
9. отправляет подходящие вакансии в Telegram;
10. сохраняет историю обработки и статистику по каждому источнику.

# Платформенная архитектура

JobMonitor строится как универсальная платформа с подключаемыми провайдерами источников вакансий.

Production-версия 0.3 использует публичные официальные ленты Habr Career, Remote OK и We Work Remotely. Версия 0.4 добавляет публичный API Remotive. Версия 0.5 добавляет выбранные публичные Greenhouse Job Boards. Версия 0.6 добавляет межплощадочную дедупликацию, сохранение происхождения вакансий и контроль очереди AI. Версия 0.7 добавляет официальный открытый API «Работа России» и инкрементальный сбор по времени изменения. Версия 0.8 добавляет официальный API Jobicy с одним фильтрованным запросом раз в шесть часов.

В дальнейшем через систему провайдеров могут быть подключены другие официально
доступные источники, например Lever. Telegram-каналы,
Kwork и Workzilla не входят в текущий план интеграций.

Каждый источник должен подключаться как отдельный провайдер без изменения основной бизнес-логики приложения.

## Доступ к источнику

Использовать официальный RSS Habr Career:

`https://career.habr.com/vacancies/rss`

Использовать официальный JSON API Remote OK:

`https://remoteok.com/api`

Использовать официальный публичный RSS We Work Remotely:

`https://weworkremotely.com/remote-jobs.rss`

Использовать официальный публичный JSON API Remotive:

`https://remotive.com/api/remote-jobs`

Публичный API Remotive публикует вакансии с задержкой около 24 часов. Это принимается как ограничение бесплатного официального источника.

Использовать публичный Greenhouse Job Board API:

`https://boards-api.greenhouse.io/v1/boards/{board_token}/jobs?content=true`

Публичные GET-запросы Greenhouse не требуют API-ключа. Каждая компания подключается отдельным провайдером и получает собственное состояние расписания. Список задаётся через `GREENHOUSE_BOARDS` в формате `token|Company Name;token|Company Name`.

Использовать официальный открытый API «Работа России»:

`https://opendata.trudvsem.ru/api/v1/vacancies`

API не требует ключа. Поисковые фразы передаются параметром `text`, регион при необходимости задаётся endpoint `region/{region_code}`, а повторные циклы используют `modifiedFrom` и время последнего успешного запуска. В уведомлении сохраняются название источника и прямая ссылка на карточку портала для атрибуции. Контактные данные работодателя из API не сохраняются.

Для Remote OK, We Work Remotely и Remotive сохранять прямую ссылку на страницу источника в каждой возможности. Для Remotive в уведомлении обязательно сохранять название источника и ссылку на Remotive.

RSS не требует пользовательской авторизации, OAuth или регистрации приложения. HTML-скрейпинг страниц Habr Career в MVP запрещён.

API Habr Career в MVP не используется: официальный API требует регистрации и активации приложения, а также OAuth 2.0. Переход на API допускается отдельным архитектурным решением в будущей версии.

Все секреты и параметры доступа хранить только в `.env`.

Если для доступа к API требуется регистрация приложения, токены, API-ключи или иные параметры авторизации, они также должны храниться только в `.env`.

## Мониторинг

Приложение должно работать непрерывно.

Основной интервал worker задаётся `CHECK_INTERVAL_SECONDS`. Для каждого источника
отдельно задаются:

- флаг включения `*_ENABLED`;
- интервал опроса `*_POLL_INTERVAL_SECONDS`.

Значения по умолчанию:

- Habr Career — 300 секунд;
- Remote OK — 900 секунд;
- We Work Remotely — 900 секунд.
- Remotive — 21 600 секунд (6 часов, не более четырёх запросов в сутки).
- каждая Greenhouse-доска — 3 600 секунд.
- «Работа России» — 3 600 секунд.
- Jobicy — 21 600 секунд (6 часов, один фильтрованный запрос за цикл).

Интервалы Habr Career, Remote OK и We Work Remotely не должны быть меньше 60 секунд. Интервалы Remotive и Jobicy не должны быть меньше 21 600 секунд, Greenhouse — меньше 900 секунд, «Работа России» — меньше 3 600 секунд. Состояние расписания хранится в
SQLite, поэтому перезапуск worker не приводит к немедленному повторному запросу
источника. Неудачная попытка также учитывается расписанием и не создаёт частых
повторных запросов к недоступной площадке.

## Нормализация данных

Независимо от источника, вакансия внутри приложения должна приводиться к единой структуре.

Минимальные поля внутренней модели:

- `source`
- `external_id`
- `title`
- `description`
- `salary_from`
- `salary_to`
- `currency`
- `published_at`
- `url`
- `company_name`
- `location`
- `remote_type` (`remote`, `hybrid`, `onsite`, `unknown`)
- `collected_at`

Эта модель должна быть одинаковой для всех будущих провайдеров.

Модель реализуется типизированным неизменяемым объектом `Opportunity` в
`models/opportunity.py`. Она проверяет обязательные поля, абсолютный URL,
диапазон зарплаты, валюту и наличие часового пояса у дат.

Каждый источник реализует единый контракт `OpportunityProvider` из
`providers/base.py` и возвращает только объекты `Opportunity`. Остальные
компоненты приложения не должны зависеть от исходного формата API, RSS или HTML.

## AI-анализ

Каждая новая вакансия отправляется в AI.

AI обязан вернуть структурированный JSON.

Обязательные поля:

- подходит / не подходит;
- AI Score (0-100);
- краткое объяснение;
- предполагаемое время выполнения;
- уровень сложности;
- возможные риски;
- краткий план выполнения;
- пример отклика работодателю.

AI не должен придумывать факты.

Если информации недостаточно, это должно быть явно указано.

Для анализа используется OpenAI Responses API со Structured Outputs и строгой
JSON Schema. API-ответ повторно валидируется локальной моделью `AIAnalysis`
перед сохранением.

### Блок 9.1. Двухнаправленный профиль и AI-поиск

Система не ограничивает сбор вакансий одной страной или одной должностью. Все
официально подключённые источники продолжают собирать доступные им возможности.
Профиль пользователя определяет не то, какие данные разрешено собирать, а то,
как AI оценивает, ранжирует и доставляет найденные возможности.

Один пользователь имеет один профиль и может включить одно или оба независимых
карьерных направления:

- `ai_automation` — автоматизация процессов, AI-инструменты, интеграции,
  боты, workflow, junior-, contract- и freelance-роли;
- `infrastructure_business_projects` — инфраструктурные и бизнес-проекты,
  развитие, управление проектами, партнёрства и смежные роли.

Направление не является отдельным Telegram-ботом, отдельной учётной записью или
отдельной базой вакансий. Оно является набором правил сопоставления и
уведомлений внутри единого профиля.

#### Структура анкеты Telegram

Анкета состоит из трёх частей.

1. **Общие условия:** предпочитаемые страны и регионы, допустимые форматы
   работы (удалённо, гибрид, офис), рабочие языки, готовность к релокации,
   юридические или часовые ограничения, общий минимальный уровень оплаты и
   общие исключения.
2. **Направление AI Automation:** позиционирование, реальные навыки, типовые
   задачи, желательные роли, допустимый уровень, предпочтительный формат
   занятости, минимальная оплата, исключения и порог уведомлений.
3. **Направление Infrastructure & Business Projects:** позиционирование,
   релевантный опыт, типовые задачи, желательные роли, география, формат
   работы, минимальная оплата, исключения и порог уведомлений.

Анкета не должна содержать более **14 вопросов**. Обязателен только выбор
направления; остальные ответы можно пропустить и позднее изменить. Большинство
ответов выбираются кнопками Telegram или множественным выбором. Свободный текст
используется только для примеров реальных задач, опыта и слов, по которым AI
должен узнавать релевантную работу. Итоговый экран подтверждения не считается
вопросом.

Анкета собирает сигналы совпадения из текста вакансии, а не абстрактные
пожелания. Утверждённый набор вопросов:

1. Какие направления включить: AI Automation, Infrastructure & Business
   Projects или оба.
2. В каких странах и регионах искать; можно выбрать «весь мир».
3. Какие форматы работы рассматривать: удалённо, гибрид, офис; можно выбрать
   все варианты.
4. Рабочие языки и готовность к релокации; ответ можно пропустить.
5. Что точно не показывать; поле необязательно и предназначено только для
   редких жёстких исключений.
6. AI Automation: какие инструменты пользователь уже применял в реальных
   задачах (например, n8n, LLM, API, CRM, Telegram, Google Workspace).
7. AI Automation: какие результаты или бизнес-задачи он уже делал либо готов
   взять на себя: сократить ручной труд, интегрировать сервисы, автоматизировать
   запись, создать бота, настроить отчётность и т.п.
8. AI Automation: какие слова и формулировки в вакансии могут означать такую
   работу, даже если там не названы конкретные инструменты.
9. AI Automation: какие роли, уровни или форматы стоит считать интересными;
   поле необязательно, вариант «показывать всё близкое» является допустимым.
10. AI Automation: какие задачи пользователь готов рассматривать как ростовую
    возможность, даже если не обладает всем перечисленным опытом.
11. Infrastructure & Business Projects: в каких типах проектов, функциях и
    отраслях есть практический опыт.
12. Infrastructure & Business Projects: какие результаты или зоны
    ответственности пользователь может взять: развитие проекта, партнёрства,
    переговоры, координация, запуск, аналитика и т.п.
13. Infrastructure & Business Projects: какие слова и формулировки в вакансии
    могут означать подходящую работу, даже если название должности другое.
14. Infrastructure & Business Projects: какие роли, уровни или ростовые задачи
    стоит показывать; поле необязательно, вариант «показывать всё близкое»
    является допустимым.

В Telegram пользователь должен иметь команды `/profile`, `/edit_profile` и
`/skills`. Меню анкеты должно позволять включать или выключать каждое
направление, редактировать только выбранный раздел и показывать итог перед
явным подтверждением. Промежуточные ответы сохраняются в `profile_sessions`,
чтобы заполнение можно было продолжить после перезапуска.

Личные сведения пользователя, ответы анкеты, ключи и токены хранятся только в
production-хранилище. В репозитории допускаются лишь обезличенные примеры
структуры данных.

#### Модель данных

Существующая таблица `user_profiles` остаётся владельцем общих условий. Для
направлений должна быть добавлена отдельная миграция SQLite с сущностью
`profile_directions`, содержащей идентификатор направления, статус включения,
версию профиля и структурированные предпочтения. Результат AI-анализа должен
хранить:

- основное рекомендованное направление;
- оценку совпадения по каждому включённому направлению;
- причины совпадения и ограничения;
- оценку доступности: формат работы, страна, язык, релокация и иные явно
  указанные условия;
- итоговое решение об уведомлении.

#### Правила AI и Telegram

AI должен анализировать каждую новую возможность по всем включённым
направлениям, а не искать одно совпадение с общим списком навыков. Для каждого
направления рассчитывается отдельный `direction_score` от 0 до 100. Итоговый
`Opportunity Score` формируется из наиболее релевантного направления с учётом
общих ограничений. AI не вправе считать международную вакансию неподходящей
только из-за отсутствия российского рынка: ограничения по стране, языку,
удалённости и праву на работу должны быть явно названы в объяснении.

`direction_score` не является самостоятельным решением и не должен показываться
пользователю как необъяснимый балл. Он строится из наблюдаемых признаков:

- совпадение ожидаемого результата и задач вакансии с примерами из профиля;
- совпадение инструментов, навыков и их смысловых аналогов в тексте;
- близость роли, отрасли или типа проекта к практическому опыту;
- наличие ростовой возможности, которую пользователь разрешил рассматривать;
- условия доступности: страна, язык, формат работы, релокация и часовой пояс.

В Telegram AI обязан показать не только число, но и категорию и доказательства:
`сильное совпадение`, `есть смысл посмотреть` или `ростовая возможность`;
что именно совпало; чего не хватает или что нужно уточнить. Числовая оценка
остаётся внутренним способом сортировки и настройки очереди.

Профиль используется прежде всего для **ранжирования, а не для жёсткого
отсечения**. По умолчанию AI должен предпочесть лишнее объяснённое уведомление
молчаливой потере потенциально интересной возможности. Жёстко не отправляются
только подтверждённые дубли, объявления с явными общими стоп-условиями
пользователя и технически неполные или недостоверные записи. Остальные
возможности получают один из уровней:

- `priority` — высокое совпадение, отправляется как приоритетная;
- `review` — частичное или неопределённое совпадение, также отправляется для
  самостоятельной оценки пользователем;
- `archive` — низкое совпадение, остаётся в истории без немедленного
  уведомления.

Начальные числовые границы этих уровней настраиваются и принимаются отдельно
при запуске Блока 9.3. Решение AI не заменяет решение пользователя: он всегда
самостоятельно открывает ссылку и оценивает вакансию.

Telegram остаётся единым каналом. Уведомление должно содержать метку основного
направления, формат работы, страну или регион, итоговую оценку и краткое
объяснение, включая причины сомнения для уровня `review`. Повторная отправка одной и той же карточки запрещена глобальной
дедупликацией независимо от направления. Сбор обратной связи в Блок 9 не
входит и будет рассматриваться отдельным этапом.

#### Границы Блока 9.1

Блок 9.1 не добавляет новые площадки, не меняет ключи, не запрашивает личные
данные без подтверждения пользователя и не изменяет поисковые запросы внешних
источников. После приёмки профиля отдельным решением может быть выполнен этап
настройки запросов источников на основе утверждённых предпочтений.

Результат содержит:

- `suitable`;
- `score` от 0 до 100;
- `summary`;
- `estimated_effort`;
- `difficulty` (`low`, `medium`, `high`, `unknown`);
- `risks`;
- `action_plan`;
- `application_draft`;
- `missing_information`.

Подтверждённый профиль кандидата хранится в таблице `user_profiles` локальной
SQLite-базы. Пользователь заполняет и изменяет его через Telegram-бота.
Промежуточные ответы анкеты сохраняются в `profile_sessions`, поэтому заполнение
можно продолжить после перезапуска. Репозиторий содержит только обезличенный
пример `profiles/user_profile.example.json` для технического импорта.

Новые записи атомарно переводятся в статус `analyzing`. Успешный подходящий
результат получает статус `analyzed`, неподходящий — `rejected`, ошибка API или
валидации — `failed`. Ошибка одной записи не должна останавливать пакет.

Ответы API не сохраняются на стороне OpenAI через параметр `store=false`.

## Telegram

Отправка подходящих возможностей выполняется отдельной командой:

`python notify.py`

Перед отправкой приложение создаёт запись в SQLite-таблице `telegram_notifications`.
Для одного `opportunity_id` допускается только одна запись доставки. Успешно отправленная
возможность получает статус `notified`, а повторный запуск не отправляет её снова.
При временной ошибке Telegram сохраняются статус ошибки и диагностическое сообщение;
следующий запуск может повторить доставку.

До начала AI-анализа бот проводит onboarding пользователя:

- `/start` — открыть меню;
- `/profile` — посмотреть подтверждённый профиль;
- `/edit_profile` — заполнить профиль заново;
- `/skills` — перейти к редактированию профиля навыков.

Анкета собирает позиционирование, реальные навыки и задачи, желаемую работу,
исключения и условия поиска. Перед сохранением бот показывает итог и требует
явного подтверждения.

В Telegram отправлять:

- источник;
- название;
- зарплату или вилку;
- AI Score;
- почему подходит;
- риски;
- план действий;
- готовый отклик;
- ссылку на вакансию.

## Защита от дублей

Повторная отправка запрещена.

Повтор одной записи на той же площадке определяется связкой:

- `source`
- `external_id`

Для MVP нормализованные возможности сохраняются в локальной SQLite-базе.
Путь задаётся настройкой `DATABASE_PATH`; значение по умолчанию —
`data/jobmonitor.db` относительно корня проекта.

Для межплощадочной дедупликации URL приводится к каноническому виду без fragment,
известных tracking-параметров, лишних слешей и различий регистра хоста. Дополнительно
строится строгий SHA-256 отпечаток нормализованных компании, названия, локации и
описания. Совпадение URL или отпечатка связывает новую запись с существующей
канонической возможностью и не возвращает её в очередь AI.

Таблица `opportunity_sources` хранит все площадки, внешние ID и исходные ссылки,
на которых была обнаружена одна каноническая возможность. Поэтому объединение
не приводит к потере происхождения данных.

Статусы обработки:

- `new`
- `analyzing`
- `analyzed`
- `notified`
- `rejected`
- `failed`

SQLite является локальным хранилищем однопользовательского MVP. PostgreSQL
остаётся за границами текущей версии.

Таблица `source_states` хранит последнее состояние каждого источника, включая
время последней попытки и успешного запуска. Таблица `source_runs` хранит историю
запусков источников со статусом и количеством полученных, сохранённых,
повторных и отложенных записей. Незавершённый запуск источника после аварийной остановки
автоматически помечается как неуспешный в следующем цикле.

## Логирование

Логировать:

- запуск;
- остановку;
- ошибки;
- найденные вакансии;
- обращения к API провайдера;
- отправку в Telegram;
- ошибки AI;
- ошибки провайдера.

Основной worker пишет журнал одновременно в консоль и в настраиваемый файл `LOG_FILE`.
Файл автоматически ротируется по `LOG_MAX_BYTES`; количество архивов задаётся
`LOG_BACKUP_COUNT`. Каждый автоматический цикл также сохраняется в SQLite-таблице
`system_runs` со статусом `running`, `completed` или `failed` и агрегированной статистикой.
Детальная статистика по каждому источнику хранится в `source_runs`, а текущее
состояние его расписания — в `source_states`.

Временные ошибки Habr Career, Remote OK, We Work Remotely, OpenAI и Telegram повторяются не более `RETRY_ATTEMPTS`
раз с паузой `RETRY_DELAY_SECONDS`. Исчерпание повторов одного источника не завершает
worker и не блокирует следующие источники или следующий цикл.

Remotive не использует быстрые повторы: на один разрешённый запуск выполняется не более одного HTTP-запроса. Следующая попытка допускается только после истечения шестичасового интервала.

# Ответственность модулей

`main.py`

Сбор и сохранение новых возможностей.

`analyze.py`

Пакетный AI-анализ записей со статусом `new`.

`profile_bot.py`

Long polling процесс Telegram-анкеты пользовательского профиля.

`worker.py`

Основной автоматический процесс. Выполняет полный цикл `сбор → AI-анализ → Telegram`
сразу после запуска, затем повторяет его через `CHECK_INTERVAL_SECONDS`. Ключ `--once`
выполняет один цикл для диагностики и приёмки.

`config.py`

Настройки приложения.

`providers/`

Изолированная работа с внешними источниками вакансий.

`models/`

Типизированная единая модель Opportunity и перечисления домена.

`services/`

Основная бизнес-логика приложения.

`utils/`

Вспомогательные функции.

`storage/`

SQLite-репозиторий, защита от дублей и статусы обработки.

`prompts/`

Промпты для AI.

# Требования к качеству

Код должен быть максимально понятным.

Каждый модуль отвечает только за одну задачу.

Не использовать глобальные переменные без необходимости.

Все ошибки должны обрабатываться.

Никаких секретов внутри исходного кода.

Проект должен запускаться одной командой Docker Compose.

Основная логика приложения не должна зависеть от конкретного провайдера вакансий.

# Definition of Done

Версия считается готовой, если:

- работает на VPS;
- работает в Docker;
- получает вакансии из официальных лент Habr Career, Remote OK, We Work Remotely, Remotive, Jobicy, выбранных Greenhouse-досок и API «Работа России»;
- AI анализирует вакансии;
- Telegram получает только подходящие вакансии;
- дубли отсутствуют;
- все настройки вынесены в `.env`;
- архитектура позволяет подключать новые провайдеры без переписывания основной бизнес-логики;
- приложение можно передать другому разработчику без изменения кода.
